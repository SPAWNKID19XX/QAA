# QAA Training App

Учебное приложение для практики автоматического тестирования.  
Стек: **Django REST Framework** (бэкенд) + **React** (фронтенд).

---

## Быстрый старт

### 1. Запуск бэкенда

```bash
cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python manage.py migrate
python manage.py seed_demo      # создаёт тестовые данные
python manage.py runserver      # http://localhost:8000
```

### 2. Запуск фронтенда

```bash
cd frontend
npm install
npm run dev                     # http://localhost:5173
```

### Авторизация

| Метод  | URL                          | Описание                        | Auth нужен |
|--------|------------------------------|---------------------------------|------------|
| POST   | `/api/auth/register/`        | Регистрация нового пользователя | Нет        |
| POST   | `/api/auth/login/`           | Логин, получение JWT-токенов    | Нет        |
| POST   | `/api/auth/logout/`          | Логаут, инвалидация refresh     | Да         |
| POST   | `/api/auth/token/refresh/`   | Обновление access-токена        | Нет        |
| GET    | `/api/auth/me/`              | Данные текущего пользователя    | Да         |
| PATCH  | `/api/auth/me/`              | Обновление профиля              | Да         |

#### POST `/api/auth/register/`

Тело запроса:
```json
{
  "username": "newuser",
  "email": "user@example.com",
  "password": "securepass123"
}
```

Ожидаемый ответ — `201 Created`:
```json
{
  "user": { "id": 5, "username": "newuser", "email": "user@example.com", "role": "user" },
  "access": "<jwt_access_token>",
  "refresh": "<jwt_refresh_token>"
}
```

Ошибки:
- `400` — username уже занят / пароль слишком простой

---

#### POST `/api/auth/login/`

Тело запроса:
```json
{
  "username": "alice",
  "password": "pass123"
}
```

Ожидаемый ответ — `200 OK`:
```json
{
  "access": "<jwt_access_token>",
  "refresh": "<jwt_refresh_token>",
  "user": { "id": 2, "username": "alice", "role": "user" }
}
```

Ошибки:
- `401` — неверные credentials → `{"detail": "No active account found..."}`

---

#### POST `/api/auth/logout/`

Тело запроса:
```json
{ "refresh": "<refresh_token>" }
```

Ожидаемый ответ — `205 Reset Content` (пустое тело)

Ошибки:
- `400` — refresh не передан / уже инвалидирован

---

#### POST `/api/auth/token/refresh/`

Тело запроса:
```json
{ "refresh": "<refresh_token>" }
```

Ожидаемый ответ — `200 OK`:
```json
{ "access": "<new_access_token>" }
```

## JWT: как работает авторизация

1. После логина в `localStorage` сохраняются два токена:
   - `access_token` — живёт 60 минут, передаётся в каждом запросе
   - `refresh_token` — живёт 7 дней, используется для получения нового access

2. Если access истёк — клиент автоматически вызывает `/api/auth/token/refresh/` и повторяет запрос

3. Если refresh тоже истёк — пользователь перенаправляется на `/login`

4. При logout — refresh-токен инвалидируется на сервере (blacklist)

## Ожидаемое поведение

### Авторизация

**Регистрация**
- При успешной регистрации сервер возвращает статус `201` и тело с полями `user`, `access`, `refresh`
- Если username уже занят — `400` с описанием ошибки в теле ответа
- Пароль короче 8 символов или слишком простой — `400`

**Логин**
- При верных данных — `200`, токены в теле ответа
- При неверных данных — `401` с полем `detail`

**Logout**
- Принимает `refresh` токен в теле запроса
- При успехе — `205`, тело пустое
- После logout refresh-токен становится недействительным: повторный запрос на `/api/auth/token/refresh/` с тем же токеном должен вернуть `401`

**Refresh**
- Обменивает refresh-токен на новый access
- Просроченный или невалидный refresh → `401`

**Защищённые маршруты**
- Запрос без токена → `401`
- Запрос с просроченным access → `401`

### Проекты

**Список проектов (`GET /api/projects/`)**
- Возвращает только проекты, где пользователь является владельцем или участником
- Admin видит все проекты
- Ответ всегда пагинирован: `{ count, next, previous, results }`
- `?status=active` — возвращает только активные проекты
- `?status=archived` — возвращает только архивные проекты
- Без фильтра — возвращает все проекты пользователя

**Создание проекта (`POST /api/projects/`)**
- При успехе — `201`, в ответе полный объект проекта
- `title` обязателен: без него → `400`
- Создавший пользователь автоматически становится владельцем

**Редактирование / удаление проекта**
- Только владелец или Admin может выполнить `PUT`, `PATCH`, `DELETE`
- Другой авторизованный пользователь получает `403`
- Несуществующий проект → `404`

**Участники проекта**
- `POST /api/projects/{id}/members/` — добавляет участника; только владелец / Admin
- Нельзя добавить владельца как участника → `400`
- Нельзя добавить уже существующего участника → `400`
- `DELETE /api/projects/{id}/members/{user_id}/` — удаляет участника; `204` при успехе, `404` если не был участником

---

### Задачи

**Список задач (`GET /api/projects/{id}/tasks/`)**
- Доступен только участникам проекта и владельцу; не-участник получает `403`
- `?status=todo` — только задачи со статусом `todo`
- `?status=in_progress` — только задачи со статусом `in_progress`
- `?status=done` — только задачи со статусом `done`
- `?priority=low` / `?priority=medium` / `?priority=high` — фильтр по приоритету
- `?assignee=<user_id>` — фильтр по исполнителю
- Фильтры можно комбинировать: `?status=todo&priority=high`

**Создание задачи**
- При успехе — `201`
- `title` обязателен
- `status` по умолчанию — `todo`
- `priority` по умолчанию — `medium`
- Если указан `assignee`, он должен быть участником проекта; иначе → `400`

**Допустимые значения полей задачи**
- `status`: `todo`, `in_progress`, `done` — любое другое значение → `400`
- `priority`: `low`, `medium`, `high` — любое другое значение → `400`

**Изменение статуса задачи (`PATCH /api/tasks/{id}/`)**
- Принимает `{ "status": "in_progress" }` и возвращает `200` с обновлённым объектом
- Недопустимое значение статуса → `400`

**Удаление задачи**
- Может удалить создатель задачи, владелец проекта или Admin
- Обычный участник (не создатель) → `403`
- При успехе → `204`

---

### Комментарии

**Получение комментариев (`GET /api/tasks/{id}/comments/`)**
- Доступно только участникам проекта, к которому относится задача
- Ответ пагинирован

**Добавление комментария (`POST /api/tasks/{id}/comments/`)**
- Поле `body` обязательно; пустое → `400`
- При успехе → `201` с объектом комментария (включая `author` и `created_at`)
- Не-участник → `403`

**Удаление комментария (`DELETE /api/comments/{id}/`)**
- Может удалить только автор комментария или Admin
- При успехе → `204`
- Другой пользователь → `403`

---

### Поиск

**`GET /api/search/?q=<строка>`**
- Ищет по `title` и `description` в проектах и задачах
- Возвращает только объекты, доступные текущему пользователю
- Ответ: `{ "projects": [...], "tasks": [...] }`
- Если параметр `q` не передан или является пустой строкой — `400` с сообщением об ошибке

### Фронтенд: ожидаемое поведение

**Логин**
- После успешного логина пользователь попадает на страницу `/projects`
- При неверных данных появляется сообщение об ошибке (элемент `login-error-msg`)
- Неаутентифицированный пользователь на защищённой странице → редирект на `/login`

**Регистрация**
- Если пароль и подтверждение не совпадают — ошибка отображается до отправки на сервер
- При успешной регистрации — редирект на `/login`

**Список проектов**
- Кнопки фильтра `All`, `Active`, `Archived` меняют список в реальном времени
- При нажатии на карточку проекта — переход на `/projects/:id`
- Пустой список показывает элемент `projects-empty`

**Страница задачи**
- Dropdown `task-status-select` позволяет изменить статус без перезагрузки страницы
- Допустимые значения в dropdown: `todo`, `in_progress`, `done`
- После добавления комментария он сразу появляется в списке
- После загрузки файла он сразу появляется в списке вложений
